# 17 July 2026: 48 kHz lossless three-lane latency

## Result

> This remains the authoritative FLAC-input record. It predates direct PCM
> fMP4 packaging and must not be used as PCM-to-PCM evidence. See the
> [raw PCM GCP and H3 capacity record](2026-07-17-pcm-h3-capacity.md) for the
> current PCM path.

Needletail carried one deterministic 48 kHz stereo FLAC AEP1 publication
through the compiled London → Amsterdam/Osaka → Tokyo relay fabric and exposed
all three delivery lanes at the Tokyo playback edge:

- native UDP with RaptorQ FEC;
- WebTransport datagrams;
- mandatory FLAC fMP4 LL-HLS over one persistent TLS 1.3/H3 connection.

The authoritative GCP run is `20260717T054206Z`. It passed clean and two-percent
loss profiles with zero missing epochs, zero missing LL-HLS parts, zero deadline
misses, and zero HLS handoff drops or worker errors. The impaired profile
recovered 43 missing source shards in every lane.

## Measurement boundary

The latency clock starts at the sample-derived publication timestamp and stops
when the receiver obtains the AEP1 epoch or LL-HLS part. The GCP sender and edge
were NTP synchronized before each profile.

The LL-HLS `estimated_render_latency_ms` field adds the configured 150 ms
receiver allowance to part availability. It is an estimate, not a measurement
of decoded audio reaching a browser, operating-system mixer, interface, or
speaker. No browser-to-speaker latency claim is made by this test.

H3 connection setup is reported separately. Steady-state init, playlist, and
part requests all used one connection; the client required TLS 1.3, negotiated
the `h3` ALPN, and verified the server certificate.

## Tested topology

| Role | Location | Zone | Machine |
| --- | --- | --- | --- |
| Contributor/origin | London | `europe-west2-b` | `e2-standard-2` |
| Primary source relay | Amsterdam | `europe-west4-a` | `e2-standard-2` |
| Warm/repair relay | Osaka | `asia-northeast2-b` | `e2-standard-2` |
| Playback edge | Tokyo | `asia-northeast1-b` | `n2-standard-2` |

The deployed route was a dual-parent DAG. Amsterdam forwarded source traffic;
Osaka supplied the independent warm/repair path. The AEP1 loss impairment was
applied at contributor ingress before the publication split, so UDP,
WebTransport, and LL-HLS qualified the same missing source data.

The route probe associated with the final deployment measured:

| Route | RTT |
| --- | ---: |
| London → Tokyo direct | 247.202 ms |
| London → Amsterdam → Tokyo | 270.375 ms |
| London → Osaka → Tokyo | 254.103 ms |

The raw UDP median of 125.480 ms is therefore close to half the measured direct
RTT. LL-HLS architecture added 3.028 ms at p50 and 9.058 ms at p99 in the clean
profile. Provider route stretch, switching, queueing, and host networking—not
part construction—account for most of the London-to-Tokyo delay.

## CPU and bounded handoff

| Profile | Contributor CPU | Edge CPU | HLS queue maximum | Capacity | Drops | Worker errors |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Clean | 81.191% | 16.915% | 7 | 4,096 | 0 | 0 |
| Impaired | 81.545% | 19.433% | 11 | 4,096 | 0 | 0 |

The contributor forwards the datagram lanes before attempting a non-blocking
enqueue to the LL-HLS worker. FLAC recovery/encoding, fMP4 construction,
playlist mutation, and canonical part publication happen asynchronously. A
full handoff increments a drop counter instead of delaying UDP or
WebTransport; the release gate requires that counter and the worker-error
counter to remain zero.

## Codec and format coverage in this historical build

At the time of this FLAC-focused run, every supported AEP1 payload kind
produced mandatory FLAC LL-HLS:

- FLAC remains FLAC;
- S16 and S24 PCM are encoded as FLAC;
- S32 and F32 PCM are normalized to S24 and encoded as FLAC;
- Opus is decoded with the pure-Rust `libopus-rs` path and encoded as FLAC.

The native and WebTransport lanes retain the original AEP1 payload. LL-HLS
uses a FLAC `fLaC` sample entry in fMP4, a separate initialization object,
standard LL-HLS playlist tags, and sample-derived PTS. Format/configuration or
PTS discontinuities start a new fMP4 continuity sequence.

## Release gates

The final GCP evidence requires all of the following:

- FLAC reaches LL-HLS in clean and impaired profiles;
- the LL-HLS client uses certificate-verified TLS 1.3/H3 and one persistent
  connection;
- UDP, WebTransport, and LL-HLS deliver every expected epoch or part;
- impaired traffic demonstrates RaptorQ recovery in every lane;
- all lane deadlines pass;
- LL-HLS queue drops and worker errors remain zero;
- queue depth remains bounded;
- wire overhead and service CPU remain within their configured budgets;
- datagram delivery remains below the deliberately buffered LL-HLS render
  estimate.

## Limits and next test

This campaign used one Tokyo playback edge. It proves the real dual-parent
relay data plane for that destination, but not simultaneous replication to
New York, Tokyo, and Sydney caches. It also does not measure browser audio
output. Those are separate qualification campaigns rather than inferred claims
from this evidence.

At the final pre-teardown audit, all four services were active, the controlled
loss chain was absent, and receiver processes had stopped. The four temporary
GCP instances, both firewall rules, all four subnets, and the qualification VPC
were then deleted. A provider-side list verified that no tagged qualification
instance, matching firewall rule, qualification network, or local lab-state
file remained.
